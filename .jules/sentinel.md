## 2025-02-28 - Secure Secret Storage in Browser Extensions
**Vulnerability:** API keys were being stored in plaintext in `chrome.storage.local`. A naive attempt to fix this was to encrypt it with a hardcoded key, or dynamically generate a key but then persist that key directly into `chrome.storage.local` alongside the encrypted payload.
**Learning:** Hardcoding cryptographic keys is a severe anti-pattern because any user can inspect the source code bundle to extract the keys. Generating a key and storing it in the *exact same* local storage dictionary as the data it encrypts provides zero extra security and constitutes security theater.
**Prevention:** For client-side data like API keys without user-supplied passwords, true robust security is impossible against an attacker with filesystem access. However, storing keys exclusively in-memory (per-session generation) raises the bar because the key isn't dumped linearly alongside the payload on disk. A production-ready approach would utilize OS-level secure enclaves or prompt the user for an encryption password upon initialization.
