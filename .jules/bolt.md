## 2025-02-15 - React state redundant updates in useSelection
**Learning:** In React, passing a structurally identical but newly created object (like a new DOMRect copy from `getBoundingClientRect()`) to `setState` will trigger a re-render because React uses `Object.is` for state comparison. This is especially problematic in events that fire rapidly (like `selectionchange`), leading to expensive layout thrashing when the actual visually useful properties (like top/left/width/height) haven't changed.
**Action:** When updating complex state objects derived from rapid events, use the functional update form of `setState((prev) => ...)` to perform a structural comparison (shallow or deep as needed) between `prev` and `next`. If they are structurally the same for your application's purposes, return the `prev` reference to explicitly bail out of the re-render.
## 2025-02-15 - React Component Redundant Re-renders in Rapid Selection Updates
**Learning:** Even if complex derived state (like `SelectionRect`) is successfully debounced and structural comparison is applied in the custom hook (`useSelection`), the React component consuming that state (`SelectionBubbleHost`) will still re-render when its *other* state updates (e.g., `status` changing from 'idle' to 'loading'). If child components (`SelectionBubble`) are not memoized, they will unnecessarily re-render and re-evaluate their DOM structure and effect closures, compounding the layout thrashing caused by rapid `selectionchange` events. Furthermore, when memoizing a child component, any callback props (like `onTrigger`) passed to it from the parent must also be memoized (e.g., via `useCallback`) to prevent the child's memoization from breaking on every parent render.
**Action:** When a parent component manages state machines and consumes frequently updating derived data, always wrap heavy presentation child components in `React.memo`. Additionally, explicitly memoize all callback props passed down to the child using `useCallback`, taking care to include all necessary dependencies (like `engine` and `targetLang`) to avoid stale closures.
