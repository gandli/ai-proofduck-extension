# 校对鸭引擎优先级与活力橙主题设计

## 目标

把校对鸭从当前的演示型处理流程，推进成可真实使用的多引擎浏览器扩展：

1. 优先使用 Chrome 内置 AI（Gemini Nano）。
2. 内置 AI 不可用时，自动切换到本地模型。
3. 本地模型优先走 WebGPU，不支持时退到 WASM。
4. 对质量要求更高时，允许切换到在线 API。
5. 翻译模式在前面几条都不可用时，允许使用第三方免费翻译兜底。
6. 界面统一升级成与品牌图标一致的活力橙主题。

## 完成标准

### 用户可见结果

1. 用户点击处理后，扩展会优先尝试 Chrome 内置 AI。
2. 如果当前环境不支持，会自动说明原因，并切到下一条可用路径。
3. 侧边栏能明确显示当前实际使用的是哪条引擎。
4. 本地模型加载时，用户能看到下载或初始化进度，而不是只看到长时间无响应。
5. 翻译、摘要、校对、润色、扩写五种模式都能走统一处理入口。
6. 界面视觉以 `#FF5A11` 为主品牌色，和当前 logo 保持一致。

### 工程结果

1. 引擎选择逻辑独立，不把所有判断堆进界面组件。
2. 各引擎实现彼此隔离，后续替换或升级不会牵一发动全身。
3. 至少覆盖编译、单测、构建和真实扩展烟雾测试。

## 总体方案

采用“统一调度层 + 独立引擎适配层 + 统一界面状态”的结构。

### 1. 调度层

新增统一调度入口，负责三件事：

1. 判断当前模式和配置应该优先走哪条路径。
2. 按优先级依次尝试可用引擎。
3. 返回统一结果，包括：
   - 实际使用的引擎
   - 结果正文
   - 给用户看的说明
   - 如果涉及模型加载，返回进度状态

### 2. 引擎适配层

引擎拆成四个独立模块：

1. `chromeEngine`
   - 对接 Chrome 内置 AI。
   - 负责能力检测、准备状态、创建会话、执行文本处理。

2. `localWebllmEngine`
   - 对接 `web-llm`。
   - 优先走 WebGPU。
   - 如 WebGPU 不可用，再使用 WASM。
   - 通过 worker 承载，避免侧边栏卡顿。

3. `onlineEngine`
   - 对接 OpenAI 兼容接口。
   - 一套配置支持 OpenAI、DeepSeek、GLM 等服务。

4. `translationFallbackEngine`
   - 仅用于翻译模式。
   - 封装第三方免费翻译接口。

### 3. 界面状态层

侧边栏不直接处理复杂判断，只消费统一状态：

1. 当前配置的首选引擎。
2. 当前实际使用的引擎。
3. 可用性说明。
4. 加载/下载进度。
5. 最终结果或错误提示。

## 引擎优先级规则

### 默认自动优先级

1. Chrome 内置 AI
2. 本地 WebGPU/WASM
3. 在线 API
4. 第三方免费翻译兜底（仅翻译模式）

### 手动切换规则

用户仍然可以在设置中手动指定首选引擎，但系统需要做两层保护：

1. 如果用户选中的引擎当前不可用，要明确说明原因。
2. 如果是翻译模式，并且手动选择的引擎失败，可以继续退到翻译兜底，避免完全不可用。

## 各引擎设计

### Chrome 内置 AI

目标是把它作为真正的默认第一选择。

实现要求：

1. 在扩展启动后检测当前环境是否支持相关 API。
2. 显示“支持 / 不支持 / 需要下载 / 准备中 / 可用”等状态。
3. 真正执行时再创建处理会话，避免无意义占用资源。
4. 不支持时给出明确原因，并自动进入下一条可用路径。

适用场景：

1. 用户想要本地优先。
2. 不想先下载大模型。
3. 希望处理速度更轻、更顺。

### 本地 WebGPU / WASM 模型

这条路径承担“Chrome 内置 AI 不可用时的主要本地能力”。

实现要求：

1. 使用 `web-llm`。
2. 模型加载放进独立 worker。
3. 优先使用 WebGPU。
4. 如果设备不支持 WebGPU，则自动退到 WASM。
5. 加载进度、模型名称、初始化状态都要能反馈到界面。

模型策略：

1. 先接一组体积和速度更可控的默认模型，而不是一开始上超大模型。
2. 模型选择不暴露过多复杂选项，先保证可用和稳定。
3. 后续如需要，再加高级模型选择。

### 在线 API

在线 API 解决“需要更强效果”的场景。

实现要求：

1. 使用 OpenAI 兼容调用格式。
2. 设置项包括地址、密钥、模型名。
3. 当用户主动指定在线模式时，优先直连在线服务。
4. 当本地链路全部不可用时，也可以自动作为下一条候选。

适用场景：

1. 用户追求更强输出质量。
2. 设备本地能力弱。
3. 本地引擎加载太慢或不可用。

### 第三方免费翻译兜底

这条路径只解决一个问题：即便前面都不行，翻译也尽量别彻底失效。

实现要求：

1. 只在翻译模式启用。
2. 优先接入实现简单、可靠的免费翻译接口。
3. 给用户明确标识当前走的是“翻译兜底”，避免误解成高质量 AI 结果。

## 界面与交互设计

### 引擎状态区

侧边栏增加更明确的引擎状态反馈：

1. 首选引擎
2. 当前实际使用引擎
3. 不可用原因
4. 加载进度
5. 当前是否已经自动回退

### 设置面板

设置面板需要从单纯配置项，升级成“引擎管理入口”：

1. 默认策略：自动优先
2. 手动指定：Chrome 内置 AI / 本地模型 / 在线 API
3. 在线 API 配置
4. 本地模型基础选项
5. 翻译兜底开关

### 活力橙主题

视觉主题统一调整为活力橙方向。

要求如下：

1. 主品牌色：`#FF5A11`
2. 按钮、选中态、焦点态、状态标签统一围绕品牌色设计
3. 保持现代、干净、清爽，不做厚重拟物风
4. 背景和卡片层次要更清楚，但不能抢内容
5. 结果区和输入区要有明确视觉分层

建议视觉方向：

1. 主按钮使用品牌橙
2. 辅助区域使用暖白、浅橙、浅灰
3. 状态标签根据就绪、加载、回退、失败使用不同层级橙色和中性色

## 代码结构调整

建议新增或重构为以下结构：

1. `entrypoints/sidepanel/lib/engine-orchestrator.ts`
   - 统一调度层

2. `entrypoints/sidepanel/lib/engines/chrome-engine.ts`
   - Chrome 内置 AI 适配

3. `entrypoints/sidepanel/lib/engines/local-webllm-engine.ts`
   - `web-llm` 本地模型适配

4. `entrypoints/sidepanel/lib/engines/online-engine.ts`
   - 在线 API 适配

5. `entrypoints/sidepanel/lib/engines/translation-fallback-engine.ts`
   - 免费翻译兜底

6. `entrypoints/sidepanel/worker.ts`
   - 本地模型 worker

7. `entrypoints/shared/contracts.ts`
   - 扩充设置、状态、消息结构

## 错误处理

需要重点处理四类失败：

1. Chrome 内置 AI 当前环境不支持
2. 本地模型加载慢或初始化失败
3. 在线 API 配置不完整或请求失败
4. 免费翻译服务请求失败

对用户的要求：

1. 说明要简单直白
2. 能自动回退时，直接告诉用户已经切到哪条路径
3. 真正无法处理时，才展示失败

## 测试策略

### 自动验证

1. 单测覆盖：
   - 引擎优先级判断
   - 回退顺序
   - 各引擎状态映射
   - 在线 API 请求与失败处理
   - 翻译兜底逻辑

2. 烟雾测试覆盖：
   - 网页选区导入
   - 侧边栏接收文本
   - 自动走默认优先级
   - 在线 API 可真实返回结果
   - 翻译兜底可触发

### 手工验证

1. 支持 Chrome 内置 AI 的环境
2. 不支持内置 AI 但支持 WebGPU 的环境
3. 不支持 WebGPU 的环境
4. 在线 API 未配置 / 已配置
5. 翻译兜底正常 / 失败

## 分阶段实现建议

### 第一波

1. 扩充设置和状态结构
2. 落地统一调度层
3. 接入 Chrome 内置 AI
4. 调整界面为活力橙主题

### 第二波

1. 接入 `web-llm`
2. 打通 WebGPU 和 WASM 回退
3. 加上加载进度显示

### 第三波

1. 完善在线 API
2. 接上第三方翻译兜底
3. 补齐测试和烟雾验证

## 本次设计结论

推荐采用“Chrome 内置 AI 优先，本地 `web-llm` 次之，在线 API 补强，翻译兜底保底”的四层结构。

这样做的优点是：

1. 完整符合“本地优先”的产品定位。
2. 用户体验更顺，不会一上来就要求复杂配置。
3. 各条引擎职责清楚，后续维护成本更低。
4. 界面主题和品牌形象可以同步成型。
